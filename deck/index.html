Start with the trust gap
The problem

AI can generate SQL and charts. People still need a reason to trust them.

“ Client context

A multi-asset trading platform is planning next month's promotion budget across crypto, gold, FX, GSS, and options.

"Which asset class should we prioritise?"

The AI can generate SQL and a chart showing crypto as the strongest completed-GTV campaign focus.

Which table? Pending transactions? Finance vs ops?

Without those checks, teams either over-trust the output or avoid using it.

The ideal interface looks like one chatbox. The real workflow needs more structure.

The LLM is the engine. The workflow is the control system that makes its output usable.

The design decision

Trust state decides the screen.

Source conflict first. Evidence next. Action last. Each trust state routes the same answer to a different product surface.

Why the workflow makes AI output safer to use

A structured interface narrows the task and exposes uncertainty.

With the right workflow, even a smaller model can be useful because the product gives it clearer jobs and safer boundaries.

Operating lesson

The product has to earn trust before the analysis can drive action.

1

Chat is powerful, but it is not the whole interface for trusted data work.

Free-form answers are fast. Decision workflows need explicit status, evidence, and warnings.

2

The workflow makes the model more useful.

Shaping, validation, and source checks reduce ambiguity before the user sees the answer.

3

The workflow transfers beyond this fintech example.

The same pattern applies anywhere AI analysis needs to become a business decision: answer, evidence, action, escalation.

Appendix B

What worked, and what did not.

The useful lesson is not that AI can make charts. It is where product structure helps AI become usable.

Worked

Keep these design choices

StateTrust state as the center. Decision-ready, evidence-needed, and blocked made the product easier to reason about.

ScreenSeparate screens by user need. Business users get action; analysts get evidence; risk gets escalation.

FallbackExplicit cached fallback. A verified cached result is useful only when it is labeled as cached.

ProcessVisual variants helped. Image-generated mockups made layout taste concrete before implementation.

Did not work

Do not rely on these

ChatChatbox-only UX. It looked simple, but it hid source choice, warnings, and next action.

ModelLetting the LLM infer everything. The model needed shaped inputs and validation boundaries.

EvidenceRaw SQL in the first screen. It made the business view feel less usable and less decision-focused.

VisualOverly decorative diagrams. If structure does not carry meaning, the slide feels polished but not persuasive.

Appendix D

Call logic: route every result to the safest next screen.

A live path is useful only when failure, cache, and audit states are explicit.

01

Shape request

POST /api/question/shape returns field quality, missing context, and recommended analysis path.

02

Run analysis

POST /api/runs starts planner, SQL agent, execution, pre-flight, and QA.

03

Route by status

Completed opens the decision pack. Audit required opens handoff. Failed never pretends to be verified.

04

Package output

CSV, email, and PPT outputs inherit the same warnings and evidence appendix.

No silent fallback.

If live validation is unavailable, the product can show a verified cached result only when it is labeled as cached and carries the original evidence.

The implementation is intentionally narrow where trust matters and flexible where users need output.
